Why this matters for you, Sriram: You've built distributed automation bots, multi-cloud ESG pipelines (Walmart, Arcelor Mittal), and invoice AI (ICE NYSE). All distributed systems. CAP is the mental model explaining why these systems behave inconsistently under failure — and every FAANG system design interview starts here.

The Three Properties

CAP theorem (Eric Brewer, 2000) states a distributed data store can only guarantee 2 of 3 simultaneously:

  • C — Consistency: Every read gets the latest write or an error. No stale data.
  • A — Availability: Every request gets a response (may be stale). Never an error.
  • P — Partition Tolerance: System operates even when network messages drop between nodes.
Critical insight: P is non-negotiable in any real distributed system — networks WILL fail (AWS AZ outages, VPN drops). So the real choice is always CP or AP — never CA.

CP vs AP — The Real Decision

Choose CP when...Banking ledgers · Payment processing · Inventory counts · Distributed locks · Config management · "Never show wrong balance"

Examples: HBase, Zookeeper, etcd, MongoDB (strong reads)
Choose AP when...Social media feeds · Shopping carts · DNS · User profiles · Recommendation engines · "Always respond, even if stale"

Examples: Cassandra, DynamoDB, CouchDB

PACELC — The Extension That Matters More

Daniel Abadi's PACELC goes beyond CAP: even when there's no partition (E), you choose between Latency (L) vs Consistency (C).

  • DynamoDB = PA/EL — available during partitions, low latency always
  • Cassandra = PA/EL — same, tunable consistency
  • CockroachDB = PC/EC — consistent always, pays latency cost
Architect's takeaway: First question for any component: "Is it worse to serve stale data, or to refuse to serve?" That decides CP vs AP. This applies to your ESG reporting (AP — stale data acceptable) vs invoice processing (CP — double-processing = disaster).

Architect Decision Framework — 3 Questions

Before designing any distributed component, ask:
1. Can my users tolerate stale data? → AP vs CP
2. Can my users tolerate downtime? → partition behaviour
3. How stale is "too stale"? → sets your replication lag SLA

Applied to Your Past Projects

  • Walmart GCP + ESG Reporting: AP — showing last week's carbon data vs this week's is fine. Prioritise availability.
  • ICE NYSE Invoice Processing (5K docs): CP — double-processing a $50K invoice is catastrophic. Use strong consistency, accept brief unavailability.
  • Apollo Azure Dashboard: AP for dashboard reads (eventual), CP for write confirmations.

Quorum Reads & Writes

Cassandra lets you tune: W + R > N guarantees consistency. With N=3 replicas: write to 2 (W=2), read from 2 (R=2) — always overlap at least 1 node with latest data. Set W=1, R=1 for max availability with weaker consistency.

Interview Framing

When asked "Design a payment system" — say immediately: "Payments are CP — I'll sacrifice availability over consistency. But the analytics dashboard and notification emails can be AP — eventual consistency is fine there." This signals senior-level thinking in the first 60 seconds.

Design a globally distributed key-value store like DynamoDB. How do you handle CAP trade-offs?
HardAmazon · Google · Meta
Show Answer ▼
Consistent hashing for key distribution, virtual nodes for uniform spread, quorum reads/writes (R+W>N), vector clocks for conflict detection, LWW for resolution, Merkle trees for anti-entropy, gossip protocol for membership. AP system, eventual consistency by default, strong opt-in. Read: Amazon Dynamo Paper →
User writes data, immediately reads it back and sees stale data. How do you diagnose and fix?
HardNetflix · Uber · Stripe
Show Answer ▼
Diagnose: replication lag (SHOW REPLICA STATUS), async vs sync replication config, read routing hitting replica before sync. Fix: (1) read-your-writes — route reads to primary for 60s post-write, (2) increase W quorum, (3) sync replication for critical paths, (4) sticky sessions for monotonic reads.
How do you design a system needing different consistency levels for different parts? (e.g. banking + notifications)
MediumRazorpay · Goldman · JPMorgan
Show Answer ▼
Tiered: (1) Ledger — CP, PostgreSQL + sync replication, serializable isolation. (2) Balance cache — eventual, Redis 5s TTL. (3) Notifications — AP, Kafka at-least-once + idempotent consumer. (4) Fraud signals — stream processing, sub-second eventual convergence.

Distributed Systems

Most Frequent
Design Twitter/X — 500M users, global, real-time feed
HardTwitter · Meta · Google
Show Answer ▼
Tweet write → Kafka → fanout service → Redis timeline cache (push for normal users, pull for celebrities with 100M+ followers → hybrid). Search: Elasticsearch inverted index. Media: CDN + S3. Ask first: DAU, tweet volume, read:write ratio. Twitter Eng Blog →
Design YouTube — video upload, processing, global streaming
HardGoogle · Netflix · Amazon
Show Answer ▼
Upload: chunked multipart to S3, resumable. Processing: async queue → transcoding workers → FFmpeg → multi-resolution. Streaming: CDN + adaptive bitrate HLS/DASH. Metadata: sharded MySQL/Spanner. Recommendations: offline ML pipeline (relevant given your IIT Madras ML track).
Design a distributed message queue like Kafka from scratch
HardAmazon · LinkedIn · Confluent
Show Answer ▼
Log-structured storage, append-only segments, O(1) writes. Partitions = unit of parallelism. Consumer groups with independent offset tracking. ISR replication, leader election via KRaft. Delivery: at-least-once default, exactly-once via transactions. Kafka Design →
Design a notification system for 1B users (push, email, SMS, in-app)
MediumMeta · Google · Amazon
Show Answer ▼
Priority queues in Kafka by urgency tier. Worker pools per channel (FCM/APNs for push, SES for email, Twilio for SMS). Rate limiting per user per channel. Delivery tracking in Cassandra. Retry with exponential backoff + dead-letter queue.
Design a URL shortener like bit.ly
MediumGoogle · Amazon · Microsoft
Show Answer ▼
Hash function (MD5/SHA256, take first 7 chars). Collision handling: try next 7 chars. Storage: key-value (DynamoDB/Redis) — short code → long URL. Redirect: 301 (cached) vs 302 (tracked). Rate limiting on creation API. Analytics: async Kafka → stream processing → ClickHouse.
